# Field Tests — Life Admin Agent

Боевые тесты для ручной проверки после деплоя.
Каждый блок — самостоятельный сценарий. Проходить в порядке нумерации внутри блока.

Два типа оценки в каждом тесте:
- **✅ / ❌** — работает / не работает (функциональность)
- **👍 / 👎 / 🤔** — нравится / не нравится / странно (UX и логика)

## FT-001: Справочник личных данных + генерация текстов

> Статус: ⬜ не проверено
> Требует: задача реализована и задеплоена

### Шаг 1 — Наполнить справочник

```
Добавь в справочник: Тойота Камри 2021, гос.номер А123БВ777
```
Ожидание: `🗂 Сохранил в справочник: Тойота Камри · #<id>`

```
Добавь в справочник: BMW X5 2023, гос.номер В456ГД777
```
Ожидание: `🗂 Сохранил в справочник: BMW X5 · #<id>`

```
Добавь в справочник: загранпаспорт РФ серия 71 №1234567, действует до 2029-05-01
```
Ожидание: `🗂 Сохранил в справочник: Загранпаспорт РФ · #<id>`

```
Добавь в справочник: паспорт Гренады №GRN789
```
Ожидание: `🗂 Сохранил в справочник: Паспорт Гренады · #<id>`

```
Добавь в справочник: домашний адрес Москва, ул. Тверская, д. 1
```
Ожидание: `🗂 Сохранил в справочник: Дом · #<id>` (или аналогичный label)

### Шаг 2 — Проверить /profile

```
/profile
```
Ожидание: список с секциями "Машины" (2), "Документы" (2), "Адреса" (1)

UX-вопросы:
- Удобно ли читать? Не слишком много полей на экране?
- Понятны ли label-ы или нужно переименовывать?

### Шаг 3 — Однозначный запрос (конкретная машина)

```
Сделай сообщение для пропуска на BMW
```
Ожидание: готовый текст с гос.номером В456ГД777, без уточняющих вопросов

### Шаг 4 — Неоднозначный запрос (две машины)

```
Сделай сообщение для пропуска на мою машину
```
Ожидание: уточняющий вопрос с нумерованным списком обеих машин

UX-вопрос: достаточно ли удобно уточнять — написать следующее сообщение с конкретным названием?

### Шаг 5 — Неоднозначный запрос (паспорта)

```
Заполни данные паспорта для анкеты
```
Ожидание: уточняющий вопрос со списком паспортов

### Шаг 6 — Обычные объекты не сломались

```
Страховка авто, истекает 1 августа
```
Ожидание: создаётся entity с кнопками, НЕ reference

### Шаг 7 — Граничный случай: "мой паспорт истекает"

```
Мой загранпаспорт истекает через год, надо продлить
```
Ожидание: создаётся entity типа document (не лезет в справочник)
Это проверка что убрали широкие intent-keywords.

## FT-002: Команды /status и /entity

> Статус: ⬜ не проверено
> Требует: в БД есть хотя бы один активный entity

### Шаг 1 — /status с несколькими объектами

Предварительно создать несколько объектов разных типов:
```
Страховка авто, истекает 1 августа
День рождения Насти 15 мая
Сертификат на массаж до 30 июня
Поездка в Стамбул 20-25 апреля
```

Затем:
```
/status
```
Ожидание: объекты сгруппированы по категориям, у объектов с близким сроком флаг ⚡

UX-вопросы:
- Читается ли за 10 секунд?
- Удобна ли группировка? Или лучше по дате?
- Нужна ли сортировка по срочности внутри группы?

### Шаг 2 — /entity с чеклистом

```
/entity <id поездки>
```
Ожидание: карточка с чеклистом (проверить документы, страховка, трансфер...)

UX-вопросы:
- Хватает ли информации в карточке?
- Не слишком ли длинно?

### Шаг 3 — /entity без аргумента

```
/entity
```
Ожидание: подсказка по использованию

### Шаг 4 — Пауза и архив

В карточке объекта нажать ⏸ Пауза → проверить /status → объект исчез
В карточке другого объекта нажать 🗄 Архив → проверить /status → объект исчез

UX-вопрос: понятно ли чем отличается "Пауза" от "Архив"? Нужны ли подписи?

### Шаг 5 — Закрыть объект кнопкой

В карточке нажать ✅ Закрыть
Ожидание: статус entity меняется на closed, напоминания отменяются
Проверить /status: объект исчез

## FT-004: Файлы (Cloudflare R2)

> Статус: ⬜ не проверено
> Требует: R2 настроен

### Шаг 1 — Обычный (searchable) PDF

Отправить любой PDF с текстом (например, договор, выписка)
Ожидание: создаётся entity, в логах Render НЕТ строки "falling back to Vision"

### Шаг 2 — Сканированный PDF

Отправить сканированный PDF (страховка, квитанция, акт)
Ожидание: в логах Render ЕСТЬ строка `PDF text too short (N chars), falling back to Vision`
Entity создаётся с правильным типом (не logistics)

UX-вопрос: правильно ли агент определил тип и название из скана?

### Шаг 3 — Фото документа

Сфотографировать страховой полис или паспорт, отправить в бот
Ожидание: OpenAI Vision извлекает данные, создаётся entity типа document

### Шаг 4 — Прикрепление файла к существующему объекту

1. Создать: `Полис ОСАГО, истекает 1 ноября`
2. Взять id из /status
3. Нажать кнопку 📎 в подтверждении
4. Отправить фото с подписью `#<id>`

Ожидание: `📎 Файл сохранён`
Затем `/entity <id>` — в карточке должен быть файл в разделе "Файлы и ссылки"

## FT-005: Качество парсинга — Entity Parser

> Статус: ⬜ не проверено
> Цель: проверить не "работает ли", а "правильно ли понимает"

### Шаг 1 — Простой документ с датой

```
Страховка ОСАГО, действует до 15 октября
```
Ожидание:
- type = document
- name ≈ "Страховка ОСАГО"
- end_date = 2026-10-15
- reminder rules: за 30, 14, 7 дней

Проверить через `/entity <id>`.

### Шаг 2 — Поездка с несколькими деталями

```
Лечу в Дубай 10 июня, обратно 17 июня, отель Marina Bay
```
Ожидание:
- type = trip
- start_date = 2026-06-10, end_date = 2026-06-17
- чеклист: документы, страховка, трансфер, check-in, чемодан
- reminder rules: за 7 дней и за 1 день

### Шаг 3 — Подарок с именем

```
День рождения Саши Иванова 22 апреля, хочу подарить что-то про спорт
```
Ожидание:
- type = gift
- name ≈ "День рождения Саши Иванова"
- end_date = 2026-04-22
- notes содержит "спорт"
- reminder rules: за 21, 10, 3 дня

### Шаг 4 — Сертификат с суммой

```
Сертификат ОЗОН на 5000 рублей, срок до конца апреля
```
Ожидание:
- type = certificate
- end_date = 2026-04-30 (или близко)
- reminder rules: за 30, 14, 7, 2 дня

### Шаг 5 — Мелкая логистика без даты

```
Купить фильтры для воды
```
Ожидание:
- type = logistics
- напоминание = digest_only (не point push)
- нет дат

### Шаг 6 — Неоднозначный ввод

```
Надо что-то сделать с машиной
```
Ожидание:
- type = logistics (fallback)
- name = что-то осмысленное
- НЕ крашится, НЕ создаёт мусор

UX-вопросы по всем шагам:
- Правильно ли определяет типы?
- Правильно ли парсит даты (особенно "конец апреля", "через неделю")?
- Чеклисты полезны или избыточны?
- Названия entities понятны или слишком длинные / короткие?

## FT-006: Кнопки — полный flow

> Статус: ⬜ не проверено
> Цель: проверить все inline-кнопки в реальных условиях

### Шаг 1 — Кнопки capture-подтверждения

Создать объект: `Сертификат в ресторан Белуга, до 31 мая`
В подтверждении нажать каждую кнопку:

**✅ OK** → ничего не происходит визуально (это нормально — только event_log)
UX-вопрос: нужна ли обратная связь после OK? Или молчание комфортно?

**📎 Добавить файл** → бот просит прислать файл
UX-вопрос: понятна ли инструкция? Нужно ли напоминать про caption #id?

**✏️ Изменить** → бот просит написать что изменить
UX-вопрос: ожидал ли ты что кнопка что-то сделает, или понятно что это заглушка?

### Шаг 2 — Кнопки напоминания

Дождаться point push (или запустить test mode) на объект с близким сроком.

**✅ Готово** → объект закрывается, напоминания отменяются
Проверить /status: объект исчез

**⏰ +7 дней** → приходит "Напомню через 7 дней"
Проверить в БД: создан новый reminder с trigger_date = today + 7

**⏰ +3 дня** → аналогично

**🙈 Игнор** → приходит "Больше не напомню"
Проверить: reminder.status = cancelled

UX-вопросы:
- Достаточно ли вариантов snooze? Нужен ли +1 день или +14 дней?
- Понятно ли чем "Готово" отличается от "Игнор"?

### Шаг 3 — Кнопки карточки объекта

Открыть `/entity <id>`, нажать все кнопки:
- ✅ Закрыть — entity.status = closed
- ⏸ Пауза — entity.status = paused
- 🗄 Архив — entity.status = archived
- 📎 Добавить файл — запрос прислать файл

UX-вопрос: все 4 кнопки нужны? Или слишком много вариантов?

## FT-008: Дайджест — качество и UX

> Статус: частично проверено (FT-003)
> Цель: оценить не "приходит ли", а "хочется ли читать каждый день"

### Подготовка

Создать набор объектов с разными сроками:
```
Страховка авто, истекает через 5 дней
Сертификат СПА, истекает через 9 дней
День рождения Насти через 12 дней
Поездка в Стамбул через 20 дней
Купить фильтры для воды (без даты)
```

Запустить test mode (SCHEDULER_TEST_MODE=true) и посмотреть дайджест.

UX-вопросы:
- Читается ли за 10 секунд?
- Правильно ли разделены "срочно" и "на этой неделе"?
- Нужны ли кнопки прямо в дайджесте или достаточно /entity <id>?
- Нужен ли объект "без даты" (купить фильтры) в дайджесте?
- Хочется ли получать его каждый день или раз в 2-3 дня?

## FT-009: Suggestion Engine — качество next_action

> Статус: ⬜ не проверено отдельно
> Цель: проверить полезность подсказок в point push

### Подготовка

Создать объект с близким сроком:
```
Страховка ОСАГО, истекает через 3 дня
```

Ожидаемое сообщение содержит:
- Название + срок

- Полезен ли next_action или очевидный?

- Нужны ли ресурсы (ссылка на сайт страховой) или это лишнее?
- Сообщение слишком длинное / короткое?

## FT-010: Фото и PDF → справочник + entity + отправка файла

> Статус: ⬜ не проверено
> Требует: задача реализована и задеплоена (r2_key, parse_and_save_reference_from_file, send_document)

### Шаг 1 — Фото документа с датой (загранпаспорт)

Отправить фото загранпаспорта, caption:
```
справочник: мой загранпаспорт
```
Ожидание:
```
🗂 Сохранил в справочник: Мой загранпаспорт · #<id>
📅 Напомню до <дата из паспорта> · /entity <id>
```

- `r2_key` заполнен в `reference_data` (через `/ref <id>` или БД)
- entity создана в `/status`

UX-вопросы:
- Понятен ли ответ бота? Достаточно информативен?

### Шаг 2 — Фото документа без даты (водительские права)

Отправить фото прав, caption:

```
справочник: водительские права
```

```
🗂 Сохранил в справочник: Водительские права · #<id>
📋 Создал запись без срока · /entity <id>
```
Проверить:
- Entity создана (через `/status`)
- Reminders не созданы (нет end_date)

### Шаг 3 — PDF (страховка/полис)

Отправить PDF страховки, caption:
```
справочник: полис КАСКО
```
Ожидание:
```
🗂 Сохранил в справочник: Полис КАСКО · #<id>
📅 Напомню до <дата> · /entity <id>
```

- Обработан как document, r2_key заполнен

### Шаг 4 — /profile после добавления

```
/profile
```
Ожидание: секция «Документы» содержит все три записи (загранпаспорт, права, полис).

### Шаг 5 — Запрос файла по имени

```
пришли права
```
Ожидание: бот отправляет файл прав прямо в чат (не ссылку).

```
пришли загранпаспорт
```
Ожидание: бот отправляет фото загранпаспорта.

UX-вопросы:
- Быстро ли приходит файл?
- Подходит ли имя файла или нужно нормализовывать?

### Шаг 6 — Запрос файла которого нет

```
пришли медицинскую страховку
```
Ожидание:

### Шаг 7 — Файл без caption (старый flow не сломан)

Отправить фото без caption → создаётся обычный entity (не справочник).
Отправить PDF без caption → то же самое.

### Шаг 8 — Машина/адрес не создают entity

```

```
